# AKILI - Parallel SpGEMM from Scratch

A step-by-step C + OpenMP implementation of Sparse General Matrix-Matrix
Multiplication (`C = A * A^T`), starting from a naive textbook version
and ending at an auto-tuned, adaptive, parallel kernel. Every step is
driven by profiling and measured data rather than guesswork, and every
result is checked against a sequential reference before it's trusted.

## Results (2 cores / 4 threads)

| Matrix | Domain | nnz(A) | nnz(C) | Config | Speedup | GFLOP/s | Correct |
|---|---|--:|--:|:--:|--:|--:|:--:|
| `cant` | FEM (cantilever) | 4.0M | 17.4M | 4T/dyn | 2.39x | 1.69 | yes |
| `bcsstk16` | Structural | 290K | 1.04M | 4T/dyn | 2.75x | 1.99 | yes |
| `ca-GrQc` | Collaboration graph | 29K | 159K | 4T/dyn | 3.22x | 0.44 | yes |
| `giant` (synthetic) | Random sparse | 500K | 5.0M | 4T/dyn | 3.63x | 0.15 | yes |
| `banded` (synthetic) | Banded | 500K | 3.9M | 4T/dyn | 5.22x | 0.35 | yes |

Speedup is best-of-3 auto-tuned parallel time against a best-of-3
sequential insertion-sort baseline (both sides measured the same way -
see the measurement-asymmetry note below). Geometric mean across all 8
benchmark matrices is 2.74x, 8/8 correct. Full tables in
[REPORT.md](REPORT.md) and [results.csv](results.csv), regenerated by
`make report`.

That single number hides some real noise. `./run_repeated.sh` reruns
the whole report N times so the noise is visible instead of hidden:
over 5 runs on this machine, mean 2.53x, stddev 0.162 (6.4%), range
2.27x-2.71x. The host has a background load average around 0.9 on 4
cores, so it's shared or virtualized rather than a quiet dedicated
box, and that shows up in the numbers. Treat any single run as an
estimate, not a fact.

## The steps

| Step | File | What it adds |
|---|---|---|
| 1 | `step1_basic_spgemm.c` | Gustavson's algorithm, CSR format, correctness baseline |
| 2 | `step2_spgemm_strategies.c` | Accumulator strategies (SPA vs hash vs heap), cache crossover |
| 3 | `step3_parallel_spgemm.c`, `step3_optimized.c` | OpenMP parallelism, pre-allocated workspaces, parallel prefix sum |
| 4 | `step4_profiler.c` | Fine-grained profiler; found that sort was 51% of the numeric pass |
| 5 | `step5_radix_sort.c` | LSD radix sort plus an adaptive per-row strategy (range-scan vs radix) |
| 6 | `step6_realworld.c` | Real SuiteSparse matrices, structure analysis, schedule chosen from CV |
| 7 | `step7_autotuner.c` | Auto-tuner: zero-run heuristic plus a bounded empirical sweep |
| 8 | `step8_report.c` | Report generator, writes REPORT.md and results.csv |
| 9 | `step9_compare.py` | Comparison against scipy.sparse, a real library instead of our own baseline |

## Step 9: how does this compare to a real library?

Every number above is our kernel against our own sequential baseline.
That tells you whether the code improved, not whether it's actually
good. Step 9 runs the identical `C = A * A^T` through scipy.sparse
(single-threaded compiled C, what most people would reach for by
default) on the same 8 matrices, same machine.

| Matrix | scipy (s) | ours (s) | ratio (scipy/ours) |
|---|--:|--:|--:|
| `1138_bus` | 0.000298 | 0.000427 | 0.70x (scipy wins) |
| `bcsstk16` | 0.047133 | 0.033025 | 1.43x |
| `ca-GrQc` | 0.003467 | 0.002063 | 1.68x |
| `cant` (see note) | 0.805461 | 0.318214 | 2.53x |
| `giant` | 0.118648 | 0.082392 | 1.44x |
| `banded` | 0.075632 | 0.031209 | 2.42x |
| `big` | 0.000471 | 0.000528 | 0.89x (scipy wins) |
| `huge` | 0.005580 | 0.003049 | 1.83x |

Geometric mean: 1.49x. That's a real edge, but a modest one, and it
flips on the two smallest matrices (`1138_bus`, `big`) where OpenMP
thread-spawn overhead costs more than scipy's single-threaded routine
ever pays. Most of the 2.74x headline number earlier is parallelism
catching up to what a mature single-threaded library already gives you
for free. On top of scipy, the extra you get from this project is
closer to 1.5x, and occasionally nothing at all.

One discrepancy came up while building this table and was worth
chasing down rather than footnoting away: `cant.mtx`'s nnz(C) doesn't
match scipy's (17,440,029 vs 17,148,509). Reimplementing the symbolic
pass in plain Python confirmed our count is the correct structural
pattern (Gustavson's algorithm: the union of columns ever touched by a
row). scipy's `csr_matmat` drops 291,520 of those entries because
their accumulated value happens to land on exact 0.0 - a real
numerical cancellation in this particular FEM stiffness matrix, not a
bug on either side. It's a convention difference (structural nonzero
count vs. stored-nonzero count) that most writeups on this topic don't
mention.

A second, smaller issue turned up the same way: `banded.mtx` was
carrying 18,375 duplicate `(row, col)` coordinate entries from an
unseeded, non-deduplicating generator. scipy sums duplicates when it
converts COO to CSR; our reader didn't. Fixed at the source, see
Reproducibility below.

`step9_compare.py` now checks the structural-nnz question automatically
for every matrix, not just `cant.mtx`. It computes scipy's normal
value-based nnz(C), and separately a "structural" nnz(C) via a
positive-only boolean matmul that can't drop cancelled entries, then
diffs that against our own count. All 8 match, which is an independent
confirmation of the symbolic phase from outside this codebase, not
just internal agreement between our own sequential and parallel paths.

## What actually went wrong along the way

- The obvious optimization was 6x slower. A range-scan "counting sort"
  looks O(n) but actually costs O(column range); for scattered rows the
  range is close to the matrix dimension, which wastes billions of
  iterations. LSD radix sort on the column list is the real O(nnz) fix,
  about 1.8x over insertion sort (Step 5).
- No single sort strategy wins everywhere. Range-scan wins on banded
  rows, radix wins on scattered rows. Picking per row by column range
  wins on both (Step 5).
- Hyper-Threading helps once there's enough work to hide the
  scheduling overhead, and hurts on tiny matrices where thread-spawn
  cost dominates (Steps 3 and 7).
- A heuristic built from reasoning alone was wrong on all 4 real
  matrices. Recalibrating it from measured data got it within 5% of
  the true optimum on the workloads that matter, which is basically
  why production libraries pair a heuristic with an empirical fallback
  instead of trusting either alone (Step 7).
- The hyperthreading/thread-count crossover was hardcoded to one CPU
  without saying so. `HT_WORK_THRESHOLD = 5,000,000` was calibrated by
  hand on the original machine and would silently have been wrong on
  anything else. It's now measured empirically on first run
  (`autotune_calib.h`) with a synthetic scatter/accumulate
  microbenchmark, and cached to `.spgemm_autotune_cache`. On the
  current machine the measured crossover came out to 200,000, 25x
  lower than the old hardcoded number. Getting a stable answer needed
  a median of 3 separate sweeps, not just repeated samples within one
  sweep, which is itself a sign this host has noisier scheduling than
  wherever the original number came from.
- AddressSanitizer and UBSan found a real heap buffer overflow in the
  parallel prefix sum (`step3_optimized.c`): when the thread count
  exceeds the row count (4 threads on a 5-row matrix, for instance) a
  chunk-boundary computation wrote one slot past the end of `rp[]`.
  Fixed by clamping the start index to `m` before the write. Every step
  now runs clean under `-fsanitize=address,undefined` on both the tiny
  test fixture and real matrices. Correctness checks against a
  sequential reference didn't catch this, because the values coming
  out were still right; only the out-of-bounds write was wrong.
- Re-testing after the calibration change surfaced a second, subtler
  bug: asymmetric measurement rigor. `step8_report.c` timed the
  parallel kernel best-of-5 but the sequential baseline only once. On
  a host with background load, that single sequential sample could
  land lucky or unlucky, and `ca-GrQc`'s reported speedup swung between
  0.52x and 3.6x run to run with no code change at all, looking like a
  regression from the lower HT threshold when it was really just a
  measurement artifact. Fixed by making the sequential baseline
  best-of-3 too; measured stability improved from 11.3% to 6.4% stddev.
  This one was only found because the numbers got re-tested after a
  change instead of trusted the first time they looked reasonable.

## Reproducibility

Every matrix used in this benchmark can be regenerated from scratch,
byte for byte, instead of existing only as a pre-built file with no
known origin:

```bash
# synthetic matrices, seeded and deterministic
python3 gen_matrix.py            # writes big.mtx, huge.mtx, giant.mtx, banded.mtx

# real matrices from the SuiteSparse collection
./fetch_matrices.sh              # writes matrices/*.mtx
```

`gen_matrix.py` originally produced only `big.mtx`, without a fixed
seed. `giant.mtx`, `banded.mtx`, and `huge.mtx` existed only as
pre-built files with no generator saved anywhere, so the reproducible
benchmark claim didn't actually hold for most of the synthetic data.
All four are now specified by size, structure, and a fixed seed in one
script.

## Build and run

```bash
make verify
```

That one command regenerates the matrices, rebuilds every step under
ASan/UBSan and smoke-tests it, runs the full benchmark report, and
runs a 5x stability check, exiting non-zero if anything fails.

To do it by hand instead:

```bash
# build any step
gcc -O3 -Wall -fopenmp -march=native -o step6 step6_realworld.c mmio.c -lm
# (or: make step6 / make all)

# fetch real matrices, regenerate synthetic ones
./fetch_matrices.sh
python3 gen_matrix.py
# (or: make matrices)

# reproduce the full benchmark report
gcc -O3 -Wall -fopenmp -march=native -o step8 step8_report.c mmio.c -lm
./step8 matrices/*.mtx giant.mtx banded.mtx big.mtx huge.mtx
# (or: make report)

# quantify run-to-run noise
./run_repeated.sh 5

# rebuild every step under ASan/UBSan and smoke-test it
# (or: make sanitize)

# reproduce the scipy comparison, needs numpy and scipy
python3 -m venv .venv && .venv/bin/pip install numpy scipy
.venv/bin/python3 step9_compare.py matrices/*.mtx giant.mtx banded.mtx big.mtx huge.mtx
```

## Techniques demonstrated

Sparse linear algebra (CSR/CSC, Gustavson's algorithm), OpenMP
parallelism (thread-private workspaces, false-sharing avoidance,
static/dynamic scheduling), cache-aware design, performance profiling
and roofline analysis, algorithm complexity in practice (O(n^2) vs
O(n) sorts), and auto-tuning that pairs a heuristic model with an
empirical search.

## Dependencies

- GCC with OpenMP
- `mmio.c` / `mmio.h`, the NIST Matrix Market I/O library (included)
- Python 3, standard library only, for `gen_matrix.py`
- Python 3 with numpy and scipy for `step9_compare.py` (Step 9 only)
